Weather Bid Command CenterV2 : AUDITÉ

Ajustement automatisé des enchères Google Ads selon la météo, construit pour Espace pour la vie (Biodôme, Planétarium, Jardin botanique, Biosphère). V2 : ajustements calculés depuis une valeur de base (fini la dérive), support PMax réel, neige et froid détectés, rotation créative météo.

Radar week-end

01

Source météo et seuils

Le script interroge les prévisions (forecast), pas la météo en temps réel : c'est la prévision qui pilote une décision prise 1 à 2 jours à l'avance. Les seuils définissent ce qui compte comme "beau temps" et "froid".

Choisir une ville remplit les coordonnées automatiquement. Les modifier à la main passe en mode personnalisé. Les coordonnées déterminent QUELLE météo pilote les enchères : mauvaise ville = mauvaises décisions. Pour trouver des coordonnées précises : clic droit sur Google Maps, le premier chiffre est la latitude. Si la ville change plus tard : la changer ici, régénérer, et remplacer le code du script dans Google Ads (ou modifier directement LOCATION_LAT / LOCATION_LON dans le script planifié).
au-dessus, sans pluie = condition "beau temps"
en dessous = traité comme "froid/pluie" (plan B intérieur)
💡 Le plan gratuit (/data/2.5/forecast, sans carte de crédit) couvre un forecast par blocs de 3h sur 5 jours : suffisant pour "demain" et pour jeudi/vendredi qui regardent samedi/dimanche. V2 : la neige, la bruine et les orages comptent maintenant comme "pluie" (essentiel pour l'hiver montréalais), et une journée sèche mais glaciale est traitée comme mauvais temps extérieur.
02

Matrice campagnes × météo

Une ligne par campagne (ou groupe de musées). NOUVEAU V2 : la colonne "Valeur de base" est la référence fixe (budget quotidien en $ ou target). Chaque ajustement est calculé depuis cette base, jamais depuis la valeur courante : aucun effet cumulatif possible, et retour automatique à la base en condition neutre.

Musée / campagne Filtre nom (contient) Sensibilité Type d'ajustement Valeur de base Palier froid/pluie Palier beau temps PMax?
La valeur de base est OBLIGATOIRE : sans elle, le script ignore la ligne (fail-safe, jamais de devinette). Une ligne couvre UN type de campagne : pour un musée qui a du Search ET du PMax, créer deux lignes avec le même filtre (une PMax = Oui, une PMax = Non). Budget = budget quotidien de référence en $. Target = tCPA en $ ou tROAS dans l'unité affichée par Google Ads. Attention au sens pour un tROAS : monter le tROAS rend l'enchère PLUS restrictive, donc un "push" pluie sur une campagne tROAS = palier négatif.
03

Anticipation week-end

Le conseil clé : la décision d'aller au musée se prend souvent le jeudi ou le vendredi. Le script regarde alors la météo de samedi/dimanche, pas celle du jour même.

05

Créatifs météo (nouveau)

Fait alterner les annonces (Search) et les asset groups (PMax) selon la météo du JOUR MÊME. Le message suit la réalité du visiteur ("Il pleut? Parfait pour le Biodôme"), pendant que les budgets, eux, anticipent la demande.

Activer la rotation créative météo
DÉSACTIVÉ par défaut : implique des pauses/activations automatiques d'annonces, à valider avec Melissa/Alexandre avant production (doctrine anti auto-pause)
Fonctionnement : jour de pluie → les annonces/asset groups "pluie" sont activés et les "soleil" mis en pause (et inversement). Condition neutre → tout est réactivé. Setup requis : créer les deux libellés dans Google Ads et les appliquer aux annonces concernées; nommer les asset groups PMax avec le marqueur (ex: "Biodôme [PLUIE]"). Une annonce ou un asset group sans libellé/marqueur n'est JAMAIS touché (evergreen).
⚠️ Garde-fou intégré : le script n'appliquera JAMAIS une pause qui laisserait un groupe d'annonces sans aucune annonce active, ni une campagne PMax sans aucun asset group actif. Dans ce cas la pause est ignorée et un avertissement est journalisé. Ordre d'exécution : activations d'abord, pauses ensuite.
06

Générer le Google Ads Script

Le script est pensé pour tourner tôt le matin (5h-7h) via le planificateur natif de Google Ads Scripts, pour que les baisses de budget prennent effet avant que la plateforme n'ait dépensé au maximum de la journée. V2 : mode Preview respecté (isPreview = aucune écriture, journal seulement).

07

Avant d'activer : checklist Alexandre

  • 01Remplir la valeur de base de CHAQUE ligne de la matrice avec les vrais budgets/targets du compte : le script ignore toute ligne sans base (fail-safe).
  • 02Vérifier la stratégie d'enchères de chaque campagne (Target CPA/ROAS = ajuster le target ; enchère manuelle ou Max Clicks = ajuster le budget) avant d'activer.
  • 03Les baisses de budget doivent être poussées tôt le matin : sinon Google Ads dépense le montant max déjà atteint dans la journée.
  • 04Sur PMax : pas de contrôle par mot-clé, mais budget de campagne, tROAS/tCPA et asset groups restent pilotables par script. Impact moins granulaire, à documenter dans le rapport client.
  • 05Documenter chaque seuil ($ ou %) avant le lancement : Alexandre veut la matrice complète en main avant l'ETA.
  • 06Garder un groupe témoin (hold-out) actif dès le jour 1 : sans lui, impossible de prouver que le script génère de la performance incrémentale plutôt que de suivre une demande qui serait arrivée pareil.
  • 07Exclure les jours fériés, vacances scolaires et événements spéciaux (Jardins de lumière, etc.) : la météo n'y explique pas la demande, le signal serait faux.
  • 08Espacer les changements de target CPA/ROAS (cooldown) : des changements trop fréquents remettent l'algorithme Smart Bidding en apprentissage permanent.
  • 09Créatifs météo : valider le principe des pauses/activations automatiques avec Melissa/Alexandre AVANT d'activer le toggle (doctrine anti auto-pause). Créer les libellés meteo-pluie/meteo-soleil et les appliquer; nommer les asset groups PMax avec [PLUIE]/[SOLEIL].
  • 10Toujours faire un Preview du script avant la première exécution planifiée : en Preview, la V2 journalise tout mais n'écrit rien.
  • 11La matrice est la SOURCE DE VÉRITÉ des bases : tout changement volontaire de budget/target (client ou équipe) doit être reflété dans la matrice, puis le script régénéré. La garde de dérive bloque les écritures quand la réalité s'éloigne trop de la base, mais la discipline reste humaine.
  • 12Sur un petit compte, préférer le hold-out manuel : appliquer le libellé wb-holdout sur 1-2 campagnes choisies (le % par hash peut tomber sur 0 campagne au hasard).
